# Decision Log

Personal Finance OS v1의 기술/설계 결정 기록. ADR(Architecture Decision Record) 형식으로 작성한다.

각 항목은 **배경 → 결정 → 이유 → 대안 → 트레이드오프** 순서로 기록하며, 상태는 `Accepted`(확정) / `Open`(미결정) / `Deferred`(보류) 중 하나로 표기한다.

---

## 0001. Asset(마스터 데이터)와 Transaction(이벤트)을 분리한다

- **상태**: Accepted (2026-07-02)
- **배경**: 종목(ZEC, 삼성전자 등) 정보와 "언제 얼마에 사고팔았다"는 사건을 어떻게 모델링할지 결정 필요
- **결정**: `Asset`(변하지 않는 마스터 정보)과 `Transaction`(매수/매도 이벤트, append-only)을 별도 엔티티로 분리한다. `Transaction`이 `Asset`을 참조(N:1)
- **이유**: 같은 테이블로 합치면 매수할 때마다 종목 정보(이름, 심볼 등)가 중복 저장됨. 마스터 데이터와 이벤트 로그는 변경 빈도와 성격이 근본적으로 다름
- **대안**: 단일 테이블에서 관리 → 기각 (중복, 정합성 깨질 위험)
- **트레이드오프**: 조회 시 조인이 하나 늘어남

## 0002. Transaction에는 체결 정보만 저장, 현재가치 환산은 실시간 조회

- **상태**: Accepted (단, 체결 시점 환율 스냅샷은 별도로 필요함 — 0013에서 최종 결정)
- **배경**: 바이낸스에서 USDT로 코인을 매수할 때, 원화환산가/김치프리미엄 등 여러 값이 동시에 존재함
- **결정**: Transaction에는 실제 체결 정보(수량, 체결통화, 체결가)만 저장. "지금 이 순간의 현재가치" 환산은 저장하지 않고 실시간 조회
- **이유**: 실시간으로 다시 구할 수 있는 값(현재가치)을 저장하면 최신성이 깨지고 중복 저장이 됨
- **대안**: 현재가치 환산까지 매 거래마다 저장 → 기각 (계속 변하는 값을 고정 저장할 이유 없음)
- **참고**: 이 결정은 "현재가치"에 한정된다. "그 거래가 체결된 정확한 순간의 환율"은 별도 문제이며 0013에서 다룬다

## 0003. 평균단가는 이동평균법, O(1)로 갱신한다

- **상태**: Accepted
- **배경**: 분할 매수/매도가 반복될 때마다 평균단가를 다시 계산해야 함. 매번 전체 Transaction을 스캔하면 성능 목표(500ms) 달성이 어려움
- **결정**:
  - 매수: `new_avg = (old_avg × old_qty + buy_price × buy_qty) / (old_qty + buy_qty)`
  - 매도: `avg_cost`는 불변, `quantity`만 감소, `realized_gain += (sell_price - avg_cost) × sell_qty`로 실현손익 누적
- **이유**: 매수/매도 이벤트 하나당 O(1) 갱신으로 끝나야 전체 Transaction 스캔을 피할 수 있음
- **대안**: FIFO(선입선출법) → 세금 계산 등에서 실무적으로 쓰이지만, v1 단순화를 위해 이동평균법으로 시작. 필요시 재검토
- **트레이드오프**: FIFO 대비 개별 매수 건의 보유기간 추적이 안 됨 (추후 세금 신고 기능 필요시 재설계 가능성)

## 0004. Holding은 파생 캐시 테이블로 저장하고, 쓰기 시점에 갱신한다

- **상태**: Accepted
- **배경**: 포트폴리오 수익률 API가 500ms 이내로 응답해야 함. 매 요청마다 Transaction 전체를 스캔해 집계하면 목표 달성이 어려움
- **결정**: `(Portfolio, Asset)`별 `Holding`(quantity, avg_cost, realized_gain) 테이블을 두고, 매수/매도 발생 시점에 즉시 갱신. 읽기 시점엔 Holding + 캐시된 시세만 읽는다
- **이유**: "쓰기 시점에 미리 계산해두고, 읽기는 캐시만 읽는다"는 패턴으로 API 응답 시간을 DB 집계 비용에서 분리
- **대안**: 매 요청마다 Transaction을 스캔해 실시간 계산 → 기각 (거래 건수가 쌓일수록 응답 시간이 선형으로 증가해 목표 위반 가능성 높음)
- **트레이드오프**: 원본(Transaction)과 파생값(Holding) 간 정합성을 갱신 로직이 항상 책임져야 함

## 0005. Currency(지불수단)와 Asset(변동성 추적 대상)을 분리한다

- **상태**: Accepted
- **배경**: USDT는 "산 대상"이 아니라 "지불수단"으로 쓰이는 경우가 많음. ZEC 같은 투자자산과 성격이 다름
- **결정**: `Currency`(KRW, USD, USDT, USDC)를 `Asset`과 별도 엔티티로 분리. v1에서는 USDT/USDC가 페깅되어 가치 변동이 없다고 가정
- **이유**: 손익을 추적해야 하는 대상과, 가치 척도로만 쓰이는 대상은 다른 개념
- **대안**: 스테이블코인도 Asset으로 취급 → 기각 (변동성이 없다고 가정하는 v1 범위에선 불필요한 복잡도)
- **트레이드오프**: 스테이블코인 디페깅(탈동조화) 리스크는 v1에서 의도적으로 무시함. 실제 디페깅 사고 발생 시 재설계 필요

## 0006. PK는 내부용 Long + 외부 노출용 UUID로 이원화한다

- **상태**: Accepted
- **배경**: 순차 증가 PK를 API에 그대로 노출하면 다른 사용자의 리소스 ID를 추측(IDOR)할 위험이 있음. 반면 UUID만 쓰면 인덱스/조인 성능에 불리
- **결정**: DB 내부 PK는 `Long`(auto-increment), API로 노출되는 식별자는 별도 `UUID` 컬럼
- **이유**: 조인/인덱스 성능(Long)과 외부 노출 시 추측 방지(UUID)를 동시에 확보
- **대안**: 전부 UUID로 통일 → 기각 (랜덤 삽입으로 인한 인덱스 성능 저하 우려, 추후 실측 필요)
- **트레이드오프**: 컬럼이 하나 늘고, API 레이어에서 UUID ↔ Long 매핑 관리가 필요

## 0007. Holding 동시성 제어는 원자적 UPDATE 문으로 해결한다

- **상태**: Accepted
- **배경**: 같은 Holding row에 대해 거의 동시에 두 개의 매수 요청이 들어오면 Lost Update(갱신 유실)가 발생할 수 있음
- **결정**: 앱 코드에서 "읽고-계산하고-쓰기"로 나누지 않고, 수량/평균단가 계산까지 포함한 단일 원자적 `UPDATE` 문으로 처리
- **이유**: PostgreSQL은 단일 UPDATE 문 실행 중 해당 row에 자동으로 락을 걸어, 동시 요청을 순차적으로 안전하게 처리함. 별도 락/재시도 코드 불필요
- **대안**: 비관적 락(`SELECT FOR UPDATE`), 낙관적 락(`@Version` + 재시도) → 저충돌 상황이고 순수 산술 연산이라 원자적 UPDATE로 충분하다고 판단. 계산 로직이 복잡해져 순수 SQL로 표현 불가능해지면 낙관적 락 재검토
- **트레이드오프**: 계산 로직이 SQL로 표현 가능한 범위를 벗어나면 이 방식은 못 씀

## 0008. Asset 마스터 데이터는 Lazy Sync(필요 시점 생성) 방식으로 확보한다

- **상태**: Accepted (2026-07-03 유니크 제약 수정)
- **배경**: 전세계 모든 종목을 미리 다 적재해둘 필요는 없음. 사용자가 실제로 거래한 종목만 알면 됨
- **결정**: 사용자가 거래를 입력할 때, 로컬에 없는 종목이면 그 순간 외부 API(한국투자증권 Open API - 국내/해외 주식, Binance API - 크립토)로 조회해 Asset 레코드를 생성. `symbol + type`에 유니크 제약을 걸어 동시 생성 시 중복 방지
- **이유**: 전량 사전 적재(Eager)는 대부분 쓰이지 않을 데이터를 미리 긁어오는 낭비
- **대안**: 배치로 전체 종목 사전 적재 → 기각. CoinMarketCap 추가 연동 → 기각 (Binance만으로 충분, YAGNI)
- **트레이드오프**: 첫 거래 입력 시 외부 API 호출 지연 발생 가능. TradingView는 공식 데이터 API 미제공으로 데이터 소스에서 제외
- **수정 이력**: 최초엔 `symbol + exchange`로 유니크 제약을 잡았으나, 같은 자산(예: ZEC)이 여러 거래소에 동시 상장될 수 있다는 점을 놓쳤음을 발견. "어느 거래소에서 거래했는가"는 Asset의 속성이 아니라 Transaction의 속성이므로 0011에서 Asset 유니크 키를 `symbol + type`으로 수정. 상장 여부 검증은 별도 테이블 없이 Lazy Sync 시점의 외부 API 응답이 그대로 담당

## 0009. 금액/수량 필드는 BigDecimal을 사용한다 (double 금지)

- **상태**: Accepted
- **배경**: `double`/`float`는 이진 부동소수점 특성상 `0.1 + 0.2 != 0.3` 같은 오차가 발생하며, 거래가 누적될수록 오차도 누적됨
- **결정**: 금액, 수량 등 모든 금융 관련 필드는 Java `BigDecimal`로 다룬다
- **이유**: 금융 서비스에서 계산 오차는 허용되지 않음
- **대안**: `double` 사용 → 기각 (정밀도 문제로 원천 배제)

## 0010. 통화/자산별 소수점 자릿수(decimal_places)는 값마다 다르게 관리한다

- **상태**: Accepted
- **배경**: KRW는 소수점이 없고, USD는 2자리, USDT/USDC 등 크립토는 6~8자리까지 필요함
- **결정**: 소수점 자릿수를 전역 상수로 통일하지 않고, `Currency`(및 향후 `Asset`)에 `decimal_places` 필드로 개별 관리
- **이유**: 통화/자산마다 실제 표현 단위가 다름. 통일하면 크립토 수량이 잘리거나 KRW에 불필요한 소수점이 붙음

## 0011. Asset 필드를 확정한다

- **상태**: Accepted
- **배경**: 주식(예: SK하이닉스/000660)과 크립토(예: ZEC/Zcash)를 같은 구조로 다루기 위한 필드 정리 필요
- **결정**:
  ```
  Asset
  - id (PK), public_id (UUID)
  - symbol          // "000660", "ZEC" — 안 변하는 canonical 식별자
  - name            // "SK하이닉스", "Zcash" — 사람이 읽는 표시용 이름
  - type            // STOCK / ETF / CRYPTO
  - market          // KOSPI/KOSDAQ/NASDAQ/NYSE — STOCK/ETF에만 의미 있음, CRYPTO는 null
  - decimal_places  // 소수점 자릿수, 심볼마다 다름
  ```
  유니크 제약: `symbol + type`
- **이유**: symbol/name 분리는 주식·크립토 공통 패턴. market은 크립토처럼 여러 거래소 동시 상장이 불가능한 주식류에만 의미가 있어 nullable로 둠 (0008 수정 이력 참고)

## 0012. 시세/환율은 "실시간 캐시"와 "시계열 히스토리"로 이원화한다

- **상태**: Accepted
- **배경**: "지금 이 순간의 최신가"(포트폴리오 현재가치 계산용)와 "기간별 수익 추이" 차트에 필요한 과거 가격 흐름은 서로 다른 목적의 데이터. Transaction 가격은 거래가 있을 때만 드문드문 존재하지만, 추이 차트는 거래 여부와 무관하게 매일의 시장 가격이 필요함
- **결정**:
  ```
  PriceQuote (실시간, Redis 캐시)      — asset_id, price, fetched_at
  PriceHistory (시계열, DB)            — asset_id, price, date
  ExchangeRate (실시간, Redis 캐시)    — currency_pair, rate, fetched_at
  ExchangeRateHistory (시계열, DB)     — currency_pair, rate, date
  ```
- **이유**: "일별 시세 업데이트 배치(100종목, 10초 이하)" 목표가 바로 History 테이블을 채우는 배치. 시계열 데이터라 TimescaleDB 확장 적용 후보
- **대안**: Redis 캐시 하나로 통합 → 기각 (Redis는 TTL로 소멸되는 최신값 저장 용도라 과거 이력 축적에 부적합)

## 0013. Transaction에 체결 시점 환율 스냅샷(fx_rate_snapshot)을 추가한다

- **상태**: Accepted (2026-07-03, 0002 재검토)
- **배경**: 0012의 ExchangeRateHistory는 배치 주기(예: 일 1회) 기준이라, 실제 거래가 체결된 정확한 순간의 환율과 다를 수 있음(특히 김치프리미엄처럼 단시간에 변동하는 값). 개별 거래의 정확한 원가(cost basis) 계산에는 배치 스냅샷만으론 부족함
- **결정**: `Transaction.fx_rate_snapshot` 필드에 체결 당시 해당 통화의 KRW 환산 환율을 기록. 체결통화가 KRW면 불필요(null 또는 1)
- **이유**: "거래 없는 날 사이의 추이"(0012)와 "이 거래 하나의 정확한 원가"는 서로 다른 문제라 하나로 대체될 수 없음
- **트레이드오프**: 과거에 체결된 거래를 나중에 수기로 입력(백필)하는 경우, 그 시점의 정확한 환율 소스가 없을 수 있음 — v1에서는 이 엣지케이스를 감수하고 진행

## 0014. Transaction 필드를 확정한다

- **상태**: Accepted
- **결정**:
  ```
  Transaction
  - id (PK), public_id (UUID)
  - portfolio_id (FK)
  - asset_id (FK)
  - currency_id (FK)     // 체결통화
  - type                 // BUY / SELL
  - quantity             // BigDecimal
  - price                // BigDecimal, 체결통화 기준 체결가
  - fx_rate_snapshot     // BigDecimal, 체결 당시 KRW 환산 환율 (0013)
  - exchange             // "Binance", "Upbit", "한국투자증권" 등
  - executed_at          // 체결 시각
  ```

## 0015. 오버매도 검증은 원자적 UPDATE의 WHERE 절 조건으로 처리한다

- **상태**: Accepted
- **배경**: 보유 수량보다 많은 매도 요청을 막아야 함. 조회 후 검증하는 방식은 0007과 동일한 동시성 문제(TOCTOU)를 재현할 수 있음
- **결정**:
  ```sql
  UPDATE holding
  SET quantity = quantity - :sell_qty, ...
  WHERE portfolio_id = :pid AND asset_id = :aid AND quantity >= :sell_qty
  ```
  영향받은 row 수가 0이면 "잔고 부족"으로 처리. 소수점 수량(예: 14.313023 ZEC)도 `>=` 비교로 동일하게 동작
- **이유**: 검증과 갱신 사이에 시간차가 없어야 동시 매도 요청에도 안전함
- **대안**: 애플리케이션에서 조회 후 검증 → 기각 (레이스 컨디션 재발 가능)

## 0016. 전량매도 후 재매수는 별도 로직 없이 기존 공식으로 처리한다

- **상태**: Accepted
- **배경**: 보유 수량이 0이 된 뒤 다시 매수하면 평균단가를 어떻게 계산할지 결정 필요
- **결정**: 별도의 "포지션 초기화/사이클 분리" 로직을 두지 않는다. Decision 0003의 매수 공식은 `old_qty = 0`일 때 자동으로 `new_avg = buy_price`가 되므로 특별 처리가 필요 없음. `realized_gain`은 사이클 구분 없이 해당 Portfolio+Asset의 전체 기간 누적값으로 유지
- **이유**: 기존 공식이 이미 이 케이스를 올바르게 처리함 (YAGNI)
- **대안**: 포지션 사이클을 감지해 별도 이력 테이블로 관리 → 기각 (현재 요구사항에 없음, "이전 매매와 완전히 분리된 수익률"이 필요해지면 재검토)

## 0017. Portfolio는 사용자당 다중 허용, "전체보기"는 저장 없이 계산된 뷰로 제공한다

- **상태**: Accepted
- **배경**: 장기투자용/코인 단타용처럼 목적별로 나누고 싶으면서도, 동시에 한눈에 전체 자산을 보고 싶어함
- **결정**: `User --(1:N)--> Portfolio`. "전체보기"는 별도 엔티티/테이블 없이, 사용자의 모든 Portfolio에 속한 Holding을 조회 시점에 합산해서 보여주는 API 응답으로 처리
- **이유**: 그릇(Portfolio)은 나누되, 집계는 읽기 시점에 계산하면 충분함. 저장 안 해도 되는 값(합계)을 미리 만들어둘 이유가 없음
- **대안**: "전체 포트폴리오"라는 별도 캐시 엔티티를 둠 → 기각 (Holding 캐시 구조만으로 충분히 빠르게 합산 가능할 것으로 예상, 실측 후 필요시 재검토)
